O desafio · The challenge

Dividir o trabalho em dois agentes cria três riscos

01

Divergência de contrato

Front e back inventam campos, endpoints e regras diferentes.

→ Contrato congelado antes da execução
02

Bloqueio em cascata

O front para de avançar enquanto espera o back ficar pronto.

→ Front usa mocks; back testa API sem UI
03

Integração tardia

Incompatibilidades só aparecem no fim, quando são caras.

→ Syncs S1–S5 com validação cruzada

Dependências · Critical path

Quatro regras que governam a ordem

01

G0 bloqueia tudo

Contrato primeiro, sempre. Sem congelar o v1, nenhuma trilha começa.

02

GA4 bloqueia A5+

A aprovação visual precede qualquer código de produção do front.

03

B2 não bloqueia A6

O front usa mocks e avança sozinho — B2 só bloqueia o sync S4.

04

Field Control é o último

B5 só entra depois do Kanban estável. O portal funciona sem a integração.

Decisões técnicas · Recomendações G0

Cinco escolhas, uma tese: coesão em TypeScript

Decisão 01 · a âncora

Back-end em NestJS

Um só ecossistema TS gera tipos compartilhados do OpenAPI e mapeia 1:1 com os módulos exigidos. FastAPI criaria fronteira de linguagem onde o projeto precisa de coesão.

02 · Contrato como código
openapi.yaml → tipos TS → mocks MSW → contract testing
03 · Tempo real
WebSocket (Socket.IO/NestJS) com fallback de polling
04 · Infra MVP
Railway/Render → AWS; S3, Postgres gerenciado, Redis/BullMQ
05 · Concorrência
Campo version; conflito → 409 e diff. Nunca sobrescrita silenciosa.
